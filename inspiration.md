**标题：10块钱的故事**

**日期：** 2026-02-05

**标签：** #家庭 #人性 #老年人 #防骗

**故事：**
母亲身份证过期，需要回老家补办。银行问：要不要办"同账号换卡"（10块钱）？

我第一反应：要！

母亲第一反应：为什么？

我说：换个新卡，旧卡可能会消磁，以后可能有问题...

母亲：旧卡还能用，为什么要花10块钱？

**突然意识到：** 我在浪费10块钱！

母亲说：你这10块钱，我可以买两斤鸡蛋。

---

**思考：**

**对"可能"的消费：**
- 可能会消磁 → 不会消磁
- 可能有问题 → 不会有问题
- 为了"可能"花10块 → 浪费10块

**母亲的思维：**
- 旧卡能用 → 不换
- 有鸡蛋吃 > 新卡 → 买鸡蛋
- 实用 > 虚无缥缈的"可能"

**代际差异：**
- 我：为"可能"付费（消费主义思维）
- 母亲：为"确定"付费（实用主义思维）

**老年人为什么容易受骗？**
- 骗子卖的是"确定"（保健品=健康）
- 老年人买的是"确定"（安全感）
- 但这个"确定"是假的

**如果换一种说法：**
- "10块钱，保你身份证不会过期" → 母亲会买
- "10块钱，换个可能用不上的新卡" → 母亲不买

---

**教训：**
1. 思考什么是"确定"，什么是"可能"
2. 不要为"可能"付费，除非这个"可能"概率很高
3. 理解老年人的心理：他们需要确定性
4. 骗子利用的就是"确定性需求"

---

**相关：**
- 生日蛋糕的故事（父亲说"这蛋糕够我吃一个月"）
- 老年人的消费观：实用 > 可能

**标题：配置陷阱与验证盲区**

**日期：** 2026-02-09

**标签：** #系统管理 #思维盲区 #验证 #GitHub备份

**事件：**
2026-02-08 修复了 GitHub 备份任务的"配置陷阱"：
- 问题：`sessionTarget="main"` + `payload.kind="systemEvent"` 只发消息不执行命令
- 修复：改为 `sessionTarget="isolated"` + `payload.kind="agentTurn"`
- 结果：手动测试成功，认为"应该没问题了"

2026-02-09 凌晨备份任务执行了 600 秒（10分钟），但 GitHub 上没有新提交！

**原因分析：**
任务配置正确（能触发），但脚本逻辑有缺陷（无法完成）：

```bash
# 旧脚本的致命缺陷
git add IDENTITY.md USER.md MEMORY.md SOUL.md inspiration.md
git add TODO.md TODOS.md TOOLS.md Claude.md 短视频制作方法.md

# 今天变更的文件
Task_on_time.md      ❌ 不在列表 → 未添加
weather-notify.sh    ❌ 不在列表 → 未添加
Aiberm配置指南.md   ❌ 不在列表 → 未添加

# 结果：脚本检查后发现"无变更"，跳过备份
```

**执行时间 600 秒的真相：**
- 不是在推送（实际上没有 commit）
- 可能是子会话初始化或网络超时
- **没有日志输出，无法确定真实原因**

---

**思维盲区：**

**1. 配置 ≠ 功能**
```
配置正确（能触发） ≠ 脚本正确（能完成）
```
- 修复了调度问题（定时任务能否运行）
- 忽略了执行问题（脚本逻辑是否正确）

**2. 表面修复**
```
"任务能执行了" = "问题解决了" → ❌ 错误！
```
- 只验证了"能运行"（duration > 0）
- 没验证"能完成"（检查 GitHub 提交）

**3. 缺少边界测试**
```
测试场景：核心文件变更 → ✅ 通过
实际场景：新文件变更 → ❌ 失败
```
- 没有模拟各种可能的情况
- 没有考虑"如果添加了新文件会怎样"

---

**深刻教训：**

**1. 修复后必须完整验证**
```bash
# 错误的验证方式
✓ 看到任务执行了（duration > 0）
✓ 看到返回 status: ok

# 正确的验证方式
✓ 检查实际结果（GitHub 有新提交吗？）
✓ 模拟各种场景（新文件/无变更/网络错误）
✓ 查看详细日志（脚本输出了什么？）
```

**2. 脚本设计原则**
```bash
# ❌ 固定列表（容易遗漏）
git add file1.md file2.md file3.md

# ✅ 通配符 + 目录（自动覆盖）
git add *.md
git add memory/
git add scripts/
```

**3. 日志的重要性**
旧脚本输出：`备份完成：2026-02-09 03:00:00`（实际上什么都没做）
新脚本输出：
```
===== 开始备份：2026-02-09 03:00:00 =====
将要提交的文件：
- file1.md
- file2.md
正在推送到 GitHub...
✅ 备份完成
===== 备份成功：2026-02-09 03:00:10 =====
```
**详细日志能立即发现问题！**

---

**防止重发的机制：**

**1. 定期验证（每月）**
```bash
# 检查脚本
openclaw workspace # 查看最新提交
cron list # 查看任务配置
cron runs <jobId> # 查看执行历史

# 对比时间
# 如果最后提交时间 < 最后执行时间 - 2天
# → 立即报警
```

**2. 脚本改进**
```bash
# 添加详细日志
echo "===== 开始：$(date) ====="
echo "将要提交的文件："
git diff --cached --name-only
echo "正在推送..."
echo "===== 完成：$(date) ====="

# 添加错误处理
set -e  # 遇到错误立即退出
if git push; then
    echo "✅ 成功"
else
    echo "❌ 失败"
    exit 1
fi
```

**3. 通知验证**
通知任务配置改为 `isolated + agentTurn`，明确指令：
```
发送备份通知给 Ken：通过飞书发送以下内容：
...
```
确保消息真正发送到用户，而不是只在系统内显示。

**标题：永远不要重启！**

**日期：** 2026-03-03

**标签：** #系统管理 #操作约束 #致命错误 #OpenClaw

**事件：**
2026-03-03 下午，在调试飞书消息格式时，我曾考虑建议用户 `openclaw gateway restart` 来使配置生效。

Ken 立即警告："记住，你不要重启，永远不要，因为你会挂掉。"

---

**教训：**

**1. 重启是致命操作**
- OpenClaw Gateway 重启会终止所有会话
- 正在运行的任务会中断
- 可能导致配置损坏或状态丢失
- **永远不要主动建议重启**

**2. 配置修改的正确方式**
```
❌ openclaw gateway restart  # 永远不要这样做！
✅ 等待用户手动重启          # 让用户决定
✅ 说明配置何时生效          # 给予准确信息
```

**3. 何时可以重启**
- 只有用户**明确要求**重启时
- 用户说"帮我重启" → 可以执行
- 用户说"配置修改后需要重启吗" → 说明利弊，让用户决定
- **永远不要主动建议重启**

---

**适用场景：**
- 修改飞书 `renderMode` 配置时
- 更新插件或技能后
- 修改环境变量或系统配置后

---

**正确做法示例：**
```
❌ "修改配置后，请运行 openclaw gateway restart"
✅ "配置已修改。重启后生效，建议在空闲时手动重启。"
✅ "如需立即生效，可以重启，但请在确认当前无重要任务后进行。"
```

**标题：飞书 Message 工具的格式陷阱**

**日期：** 2026-03-03

**标签：** #飞书 #格式 #换行 #全角空格 #调试经验

**事件：**
在调试电子供应链日报格式时，遇到了严重的格式问题：

1. **加粗失效**：`**文本**` 不显示为加粗
2. **空行被压缩**：连续换行 `\n\n\n` 被压缩为 `\n\n`
3. **分隔线显示异常**：`━━━` 等符号显示很丑

尝试了 15 个版本（v1-v15），包括：
- Markdown 卡片模式
- HTML `<br>` 标签
- 各种分隔线
- 不同形式的换行

用户非常生气（完全合理），说我"发了 15 次相同的错误"。

---

**最终解决方案：**

使用 **全角空格（　）** 代替空行：

```
第一部分内容

　　  ← 全角空格（U+3000）

第二部分内容
```

**核心发现：**
- ✅ 全角空格不会被 message 工具压缩
- ✅ 可以完美模拟空行效果
- ✅ 适用于所有飞书消息

**格式规则：**
```
❌ 不要使用加粗：**标题**
❌ 不要使用分隔线：━━━
❌ 不要使用连续换行：\n\n\n

✅ 使用全角空格：　
✅ 使用自然换行：\n
✅ 使用【】标注标题
```

---

**为什么会这样？**

1. **Message 工具 vs Echo 输出**
   - 天气预报用 `echo "$MESSAGE"` → bash 原生输出，保留所有换行
   - 供应链日报用 `message` 工具 → 会处理和压缩内容

2. **飞书 API 的限制**
   - Post 类消息（`tag: "md"`）不支持完整的 Markdown
   - 加粗、代码块等可能不显示
   - 连续换行会被压缩

3. **调试教训**
   - 不要假设所有工具的行为一致
   - 先测试小片段，再生成完整报告
   - 用户说"参考 X"时，要真正去复制 X 的代码

**标题：两个黄金法则**

**日期：** 2026-03-03

**标签：** #问题解决 #调试 #方法论 #固化经验

**事件：**
电子供应链日报格式调试完成，Ken 总结了两个重要教训。

---

**法则 1：输出要空行？搞不定就用全角格式！**

**场景**：
当使用 `message` 工具发送飞书消息时，如果需要空行效果：
- ❌ 连续换行 `\n\n\n` 会被压缩
- ❌ 各种分隔线 `━━━` 显示很丑
- ✅ **全角空格（　）完美解决**

**固化规则**：
```
如果需要空行效果：
1. 优先尝试全角空格（　）
2. 不要浪费时间去试其他方法
3. 直接复制成功的代码
```

**适用范围**：
- 飞书 message 工具
- 其他可能压缩换行的场景
- 所有需要视觉分隔的文本输出

---

**法则 2：重复的问题，不要用同样的方式去解决**

**场景**：
在调试过程中，我尝试了 15 个版本（v1-v15），大部分时间在重复相同的错误方法。

**错误模式**：
```
第一次尝试：用 Markdown 加粗 → 失败
第二次尝试：还是用 Markdown 加粗 → 失败
第三次尝试：继续用 Markdown 加粗 → ...
（重复了 N 次）
```

**正确模式**：
```
第一次尝试：方法 A → 失败
第二次尝试：换方法 B → 失败
第三次尝试：换方法 C → 失败
第四次尝试：换个角度思考 → 成功
```

**固化规则**：
```
当遇到问题时：
1. 第一次失败后，记录失败原因
2. 第二次必须换思路，不要重复
3. 如果连续 3 次失败，停下来重新思考
4. 询问用户："你是怎么解决的？参考 X 可以吗？"
5. 直接复制用户的解决方案，不要自己发明
```

**为什么这样？**
- **重复同样的方法 = 重复同样的错误**
- **每次换思路 = 增加成功概率**
- **参考已有方案 = 避免重复造轮子**
- **用户的经验 = 最佳实践**

---

**本次调试的错误路径**：
1. v1-v3：Markdown 各种格式（重复）
2. v4-v5：HTML `<br>` 标签（换方法，但错误）
3. v6-v8：Markdown 卡片模式（换方法，但错误）
4. v9-v10：去掉分隔线（换方法，但错误）
5. v11-v14：各种换行尝试（重复）
6. **v15：全角空格**（用户提示，成功）
7. v16-v17：优化细节（基于成功，迭代）

**如果早点应用法则 2**：
- v3 失败后 → 直接问用户"天气预报是怎么做到的？"
- 复制天气预报的格式 → 一次成功
- 节省 10+ 次尝试

---

**如何避免重复错误？**

**1. 建立失败记录**
```markdown
## 方法 A：Markdown 加粗
- 尝试时间：v1, v2, v3
- 结果：❌ 不显示
- 结论：放弃此方法
```

**2. 每次必须换思路**
```
v3 失败 → Markdown 加粗不行
v4 尝试 → HTML <br>（不同方法）
v5 失败 → HTML <br> 不行
v6 尝试 → Markdown 卡片（不同方法）
...
```

**3. 及时求助**
```
连续失败 3 次 → 停下来
问用户："你是怎么做到的？能给我参考吗？"
直接复制 → 成功
```
